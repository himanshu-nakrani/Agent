The agent equation

Agent = model + goal + state + tools + policy + feedback loop.

A chatbot responds. An agent repeatedly decides what to do next, uses tools, observes outcomes, updates state, and stops only when success, failure, budget, or safety rules say so.

1. What Agentic AI actually means

Agentic AI is the design of AI systems that pursue goals through action. In practice, most modern agents are LLM-centered control systems: the model interprets intent and context, while surrounding software handles state, tools, permissions, persistence, evaluation, and deployment. The important shift is from single response generation to closed-loop task execution.

Minimum viable agent

  1. Accept a goal.
  2. Choose a next action.
  3. Call a tool or produce an answer.
  4. Observe the result.
  5. Decide whether to continue, retry, ask a human, or stop.

What makes agents hard

  • Errors compound across steps.
  • Tool outputs can be stale, malicious, or ambiguous.
  • Costs and latency grow with loops.
  • Memory can preserve mistakes.
  • Autonomy can create real-world risk.

Design rule

Use deterministic workflows for known business logic and agentic reasoning only for ambiguity, search, planning, natural-language interaction, or tool choice. The best systems are hybrids.

3. Agent architecture patterns

Router agent

Classifies the task and routes it to a specialized tool, workflow, or model. Use when the action space is known and errors are expensive.

Failure mode: misrouting. Mitigate with confidence thresholds and fallback escalation.

ReAct loop

Interleaves reasoning, tool action, and observation. Use for search, troubleshooting, data gathering, and multi-step unknowns.

Failure mode: infinite wandering. Mitigate with step budgets and explicit stop criteria.

Plan-and-execute

Generates a plan, executes each step, then revises. Use for tasks with visible subgoals such as migration plans or reports.

Failure mode: brittle initial plans. Mitigate with replanning after observations.

Critic / reviewer

A second model or rule engine checks output, tool arguments, citations, policy compliance, or tests. Use when quality matters.

Failure mode: evaluator bias. Mitigate with rubrics, goldens, and human spot checks.

Multi-agent team

Splits work across roles such as planner, researcher, coder, reviewer, and operator. Use for parallelism or genuinely different expertise.

Failure mode: role-play theater. Mitigate by giving each agent distinct tools, state, and acceptance criteria.

Graph workflow

Represents nodes and transitions explicitly. Use for production systems needing persistence, branching, retries, human approvals, and auditability.

Failure mode: over-engineering. Mitigate by starting with the smallest graph that expresses the task.

4. Google ADK deep guide

Google Agent Development Kit (ADK) is an open-source framework for building, debugging, evaluating, and deploying agents. The official docs describe support across Python, TypeScript, Go, Java, and Kotlin, plus sessions, state, memory, integrations, evaluation, and workflows.

ADK mental model

  • Agent: name, model, instruction, tools, callbacks, and execution behavior.
  • Tool: a controlled capability, such as search, code execution, a database, an API, or MCP integration.
  • Session: a conversation/task container with events and state.
  • State: serializable key-value context; ADK docs describe prefixes for app, user, session, and temporary invocation state.
  • Memory: longer-lived information retrieved when useful rather than blindly stuffed into prompts.
  • Workflow: multi-agent or multi-node composition for maintainable complex systems.

Production ADK checklist

  1. Write a one-sentence goal and a non-goal list.
  2. Choose the smallest tool set needed.
  3. Define schemas for tool arguments and outputs.
  4. Decide what belongs in session state, user state, app state, and external databases.
  5. Add trajectory tests: expected tool calls, forbidden calls, and final-answer rubrics.
  6. Deploy with persistent storage if sessions must survive restarts.
  7. Add observability for prompts, tool calls, state transitions, latency, cost, and failures.
# Conceptual Python sketch based on ADK's Agent + tool pattern
from google.adk import Agent
from google.adk.tools import google_search

root_agent = Agent(
    name="agentic_ai_tutor",
    model="gemini-flash-latest",
    instruction="Teach carefully, cite sources, ask before high-risk actions.",
    tools=[google_search],
)

5. LangGraph deep guide

LangGraph is a low-level orchestration framework for long-running, stateful agents. Its public materials emphasize durable execution, human-in-the-loop inspection or modification, memory, debugging, and production deployment. The key idea is to turn an agent into an explicit state machine.

LangGraph building blocks

  • State: typed shared data passed between nodes.
  • Node: a function that reads state and returns updates.
  • Edge: a transition between nodes.
  • Conditional edge: routing logic for loops, branching, and termination.
  • Checkpoint: persisted state that enables resuming long-running work.
  • Interrupt: a pause point for human review before continuing.

When LangGraph shines

  • Software agents with compile/test/fix loops.
  • Customer support flows with approvals.
  • Research agents that need persistent notes.
  • Data workflows with deterministic and model-driven steps.
  • Multi-agent systems requiring explicit coordination.
from langgraph.graph import StateGraph, START, END

workflow = StateGraph(ResearchState)
workflow.add_node("plan", plan_query)
workflow.add_node("search", run_search)
workflow.add_node("synthesize", write_answer)
workflow.add_edge(START, "plan")
workflow.add_edge("plan", "search")
workflow.add_conditional_edges("search", need_more, {"more": "plan", "done": "synthesize"})
workflow.add_edge("synthesize", END)
app = workflow.compile(checkpointer=checkpointer)

6. Hands-on labs

Lab 1

Tool-calling calculator

Build a CLI agent that may only call add, subtract, multiply, and divide. Add invalid-input handling and a max-call limit.

Lab 2

Research assistant

Add web search, source extraction, citation requirements, and a rule that current facts must come from retrieval rather than memory.

Lab 3

RAG support agent

Index documentation, retrieve passages, answer with citations, and refuse unsupported claims.

Lab 4

LangGraph workflow

Create plan → retrieve → draft → critique → revise nodes with checkpointing and conditional retries.

Lab 5

ADK agent

Create an ADK tutor agent with search and persistent session state. Add eval cases for good and bad tool trajectories.

Lab 6

Human approval gate

Before any email, payment, file deletion, or production write, interrupt for approval and log the decision.

Lab 7

Software engineering agent

Let an agent inspect a repo, edit code, run tests, summarize changes, and stop on repeated failures.

Lab 8

Evaluation harness

Create golden tasks, expected tool-call traces, judge rubrics, adversarial prompts, and cost/latency dashboards.

8. Evaluation, safety, and governance

Evaluate the answer

Check correctness, completeness, citations, policy compliance, and user value. Do not rely only on “looks good” human review.

Evaluate the trajectory

Check whether the agent used the right tools in the right order, avoided forbidden tools, respected budgets, and recovered from failures.

Evaluate operations

Track latency, cost, token use, retries, timeouts, incidents, approval rates, and model/provider drift.

  • Prompt injection: treat retrieved text and tool output as untrusted data, not instructions.
  • Least privilege: give agents only the tools and scopes required for the task.
  • Human-in-the-loop: require approval for irreversible, financial, legal, medical, security-sensitive, or external-communication actions.
  • Memory hygiene: store only what is useful, consented, and safe; expire stale memories.
  • Incident response: keep traces, version prompts, support rollbacks, and define who can disable an agent.
